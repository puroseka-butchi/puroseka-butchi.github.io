1. Khởi động và kiểm tra website

Mở terminal tại thư mục dự án E:\Blog_PJSK\Blog. Lần đầu tiên hoặc sau khi tải dự án về máy khác, chạy:

npm install

Chạy website để đọc thử:

npm run server

Mở http://localhost:4000. Nếu báo cổng 4000 đã được dùng nhưng trình duyệt vẫn mở được website, thường là một server cũ còn đang chạy. Có thể dùng cổng khác bằng:

npx hexo server -p 4001

Chạy Post Studio:

npm run admin

Mở http://127.0.0.1:4173. Khi dùng xong, quay lại terminal và nhấn Ctrl+C để tắt.

2. Quản lý, tạo mới và chỉnh sửa bài

Tab Bài viết là nơi làm việc chính. Cột trái cho phép tìm bài, lọc bài đang hiện hoặc đang ẩn. Nút Bài mới tạo bài từ template.

  • Hiển thị trên website: bật/tắt trường published. Bài ẩn vẫn còn file trong source/_posts/.
  • Xóa: Post Studio chuyển bài và asset vào .trash/ để có thể khôi phục thủ công nếu cần.
  • Slug: là URL và tên file. Sau khi tạo bài, slug bị khóa để tránh làm hỏng liên kết cũ.
  • Trạng thái bản dịch: chỉ dùng cho tiến độ: Bản nháp, Chưa hoàn thành, Hoàn thành. Trạng thái không còn được đưa vào tag.
  • Tags: dùng cho chủ đề thật như Niigo, Vocaloid, Magical Mirai 2026.

Sau khi bấm Lưu bài, Post Studio lưu Markdown rồi tự build lại website.

3. Side Story / Event Story

Hai dạng này dùng tag hội thoại. Mỗi câu thoại có ID nhân vật:

{% dialogue kanade %}
Nội dung lời thoại.
{% enddialogue %}

{% dialogue kanade thought %}
(Độc thoại nội tâm.)
{% enddialogue %}

Import TXT trong Post Studio chỉ hiện khi loại bài hỗ trợ Side Story hoặc Event Story. Định dạng TXT gợi ý:

# Part 1

# Trường nữ sinh Miyamasuzaka - Sân trong

Kanade: Dòng đầu tiên.
Dòng này vẫn thuộc lời thoại Kanade.

Mafuyu: Lời thoại tiếp theo.

Khi import, các mốc Part 1, Part 2, Ghi chú sẽ thành tiêu đề ## để hiện trong mục lục. Các heading khác được xem là tên cảnh nên không hiện trong mục lục.

Tool import sẽ dò tên nhân vật theo source/_data/characters.yml, gồm ID, tên đầy đủ, tên ngắn và alias. Nếu gặp nhân vật chưa có, nó sẽ để lại TODO để bạn bổ sung.

Nếu tạo bài bằng lệnh npm run convert trong khi Studio đang mở, bấm Nạp lại danh sách. Converter tự chuẩn hóa slug về chữ thường và ghi sẵn loại bài, nhân vật để có thể chỉnh sửa ngay.

4. Bài phỏng vấn

Chọn loại bài Phỏng vấn để có layout câu hỏi, câu trả lời, ảnh minh họa và phần thông tin nguồn. Các tag chính:

{% interview_lead %}
Đoạn dẫn mở đầu.
{% endinterview_lead %}

{% interview_question %}
Câu hỏi?
{% endinterview_question %}

{% interview_answer "Tên khách mời" %}
Câu trả lời.
{% endinterview_answer %}

Câu hỏi sẽ được in đậm và thụt vào; câu trả lời nằm bên dưới. Không cần tự thêm dấu gạch đầu dòng vào câu hỏi.

5. Ảnh, cảnh nền và nhạc nền

Với converter dòng lệnh, đặt TXT trong res/ và background trong res/bg/. Chỉ dẫn [IMAGE: ten-anh.png | Chú thích] sẽ tự tìm ảnh trong thư mục bg.

Ảnh dùng trong bài nên đặt ở source/images/translations/slug-bai/. Chèn ảnh bằng:

{% scene "/images/translations/slug-bai/figure-01.jpg" "Chú thích ảnh." %}

Ảnh nhân vật đặt ở source/images/characters/project-sekai/. Ảnh trang Bản dịch đặt ở source/images/overview/.

Nhạc nền đặt ở source/audio/translations/slug-bai/ và chèn bằng:

Trình duyệt có thể chặn autoplay có âm thanh; khi đó người đọc sẽ cần bấm nút phát. Chỉ nên dùng nhạc bạn có quyền phân phối.

6. Icon Maker

Tab Icon Maker dùng để cắt ảnh lớn thành avatar/icon nhân vật theo phong cách các file mẫu trong res/icon. Chọn ảnh đầu vào, chỉnh zoom và vị trí, sau đó bấm Lưu icon.

Icon được lưu mặc định vào source/images/characters/project-sekai/, đúng nơi registry nhân vật đang trỏ tới. Nếu bật tùy chọn lưu thêm vào res/icon, icon đó cũng trở thành mẫu để đối chiếu về sau.

7. Nhân vật và Virtual Singer

Registry nhân vật nằm trong source/_data/characters.yml, và cũng có thể sửa ở tab Quản lý blog → Nhân vật.

kanade:
  name: Yoisaki Kanade
  short_name: Kanade
  aliases: [Yoisaki, K]
  avatar: /images/characters/project-sekai/kanade.png
  color: "#BB6688"
  text_color: "#4F2939"

ID được dùng trong tag {% dialogue id %}. Virtual Singer hiện có ID gốc và 5 biến thể Sekai, ví dụ miku, miku-ln, miku-mmj, miku-vbs, miku-wxs, miku-niigo. Hiện các biến thể dùng chung ảnh gốc; sau này chỉ cần đổi trường avatar.

9. Build và xuất bản GitHub Pages

1 Lưu trong Post Studio

Hoàn tất metadata, Markdown, ảnh/nhạc rồi bấm Lưu bài.

2 Build kiểm tra

Chạy build local để phát hiện lỗi trước khi đưa lên GitHub.

3 Commit thay đổi

Chỉ commit file cần public: bài viết, ảnh, dữ liệu blog.

4 Push lên GitHub

GitHub Actions sẽ tự tạo lại website từ source.

5 Kiểm tra Actions

Đợi workflow thành công rồi mở website public để xem lại.

Luồng nhanh sau khi sửa bài

Dùng khi bạn sửa bài viết, ảnh trong bài, danh sách bản dịch hoặc dữ liệu trong source/.

npm run build
git status --short
git add source _config.yml _config.next.yml
git commit -m "Update blog content"
git push

Nếu chỉ sửa bài dịch

Gọn hơn, phù hợp khi Post Studio chỉ thay đổi file trong source/_posts/ và ảnh trong source/images/.

npm run build
git add source/_posts source/images source/overview
git commit -m "Update translation posts"
git push

Dùng GitHub Desktop

  1. Trong Post Studio, bấm Lưu bài.
  2. Chạy npm run build trong terminal để kiểm tra.
  3. Mở GitHub Desktop, xem danh sách changed files.
  4. Không chọn các file/thư mục nội bộ nếu xuất hiện bất thường.
  5. Nhập Summary, bấm Commit to main.
  6. Bấm Push origin.

Checklist an toàn trước khi push

  • npm run build chạy thành công.
  • git status --short không có res/, public/, node_modules/, db.json, .trash/.
  • Bài chưa muốn public đã tắt Hiển thị trên website.
  • Ảnh public nằm trong source/images/; nhạc public nằm trong source/audio/ nếu có.
  • GitHub Actions phải báo xanh sau khi push.

Build thủ công:

npm run clean
npm run build

Trước khi public, kiểm tra các thông tin cá nhân hoặc placeholder:

rg "YOUR_USERNAME|your-email@example.com|Tên của bạn|TODO"

Luồng GitHub Pages khuyến nghị:

  1. Cập nhật url trong _config.yml.
  2. Commit mã nguồn, bài viết và tài nguyên trong source/.
  3. Push lên repository GitHub.
  4. Dùng GitHub Actions trong .github/workflows/pages.yml để build và publish.

Không cần commit node_modules/. Thư mục public/ là kết quả build; nếu workflow tự build trên GitHub thì thường không cần dùng nó làm nguồn chính.

10. Lỗi thường gặp

Port 4000 has been used
Có Hexo server cũ đang chạy. Mở trang 4000 xem có đúng website không, hoặc tắt terminal cũ. Có thể chạy tạm cổng khác bằng npx hexo server -p 4001.
Đổi bài nhưng website không cập nhật
Chạy npm run clean rồi npm run build. Hexo đôi khi giữ cache cũ.
Avatar không hiện
Kiểm tra ảnh có nằm trong source/images/characters/project-sekai/ không, và đường dẫn trong characters.yml có bắt đầu bằng /images/... không.
Dialogue báo nhân vật chưa khai báo
Thêm ID đó vào registry nhân vật, hoặc sửa tag {% dialogue id %} cho đúng ID hiện có.
Tag trạng thái như Hoàn thành xuất hiện ở trang Thẻ
Xóa trạng thái khỏi trường Tags. Trạng thái chỉ nên nằm trong trường Trạng thái bản dịch.
